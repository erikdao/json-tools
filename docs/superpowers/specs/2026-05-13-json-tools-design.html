1 · Overview & goals

A small, fast, beautiful collection of client-side JSON tools hosted at jsontools.erikdao.com. Four tools share one workspace: Beautify, Minify, Validate, Parse (interactive tree explorer).

Constraints:

  • Open to anyone — no auth, no rate limits.
  • Static-only — Cloudflare Pages, no Workers, no KV.
  • All processing in the browser; nothing is uploaded.
  • Bundle target < 150 KB gzipped for the critical path.
  • Handles payloads up to ~5 MB without main-thread strain.

Success looks like: paste anything reasonable, get the result in under 250 ms, and the page is satisfying enough to bookmark. No ads. No cookies. No upsell.

3 · Architecture & deployment

Astro + island hydration

Astro ships zero JS by default for static content (header, footer, marketing copy, per-tool intro). The single client island is the workspace component, hydrated with client:load. Critical-path JS is dominated by CodeMirror 6 (~80 KB gz).

Routes

Four tool routes share one layout; / serves a redirect. The active tool determines (a) the middle-column action button, (b) the options bar, and (c) whether the right pane is a CodeMirror output or the TreeExplorer:

  • / → server redirect to /beautify (single canonical URL, no duplicate content)
  • /beautify · /minify · /validate
  • /parse (right pane = TreeExplorer)

Tab strip uses real <a> links for SEO and progressive enhancement; client-side navigation swaps the active tool without a full reload (Astro View Transitions).

4 · Components & UX

Workspace island

Single Preact component, mounted with client:load. State owner of:

  • Input string, output string, current tool, per-tool options, error info.
  • Two CodeMirror 6 instances — input (editable) and output (readOnly). Custom theme matches the terminal palette; JSON language pack provides syntax highlighting and lint markers.
  • Action column (middle pane): vertical action button labeled by current tool (BEAUTIFY ▶, etc.). Triggered by click or ⌘/Ctrl + Enter.

When the route is /parse, the right CodeMirror is swapped for the TreeExplorer component.

TreeExplorer (the /parse tool)

Custom, not a library — keeps the terminal aesthetic intact and is ~200 LOC. Behavior:

  • Renders parsed value as nested rows; each container has an expand/collapse caret.
  • Each row: type chip · key · color-coded value · ⎘ copies the JSON path (e.g. data.users[0].email).
  • Search input at top of pane filters by key or value substring; matches highlight, ancestors auto-expand.
  • Type chip palette: obj · arr · str · num · bool · null in muted amber.
  • Virtualization kicks in above 1,000 rendered rows (rare); below that, a flat DOM tree is simpler and more reliable.

Options bar (bottom of workspace)

ToolBar contents
/beautifyIndent segmented 2 · 4 · tab · Sort keys checkbox
/minifyMuted text: › minify · no options
/validateAfter run: › valid · 4 keys · depth 3 · 342 B or error message in red with [ jump ] link
/parseMuted text; search input lives inside the explorer pane

Option values reflect in the URL as query params (?indent=tab&sort=1) so links can be preconfigured.

Header & footer

  • Header: jsontools_ wordmark (blinking underscore, respects prefers-reduced-motion) · theme cycle button (auto → light → dark) · Source ↗ GitHub link.
  • Footer (single muted line): built by erik · no tracking · v0.1 · source ↗.

Mobile (< md breakpoint, 768 px)

  • Panes stack vertically: input → action row → output (or TreeExplorer).
  • Action button becomes a full-width horizontal row.
  • Tab strip becomes horizontally scrollable.
  • Drag-drop still works (touch devices generally don't trigger it, but the file picker handles their case).

5 · Data flow

Lifecycle of one operation

  1. Input arrives via paste / typing, file picker, or drag-drop → all funnel into setInput(string).
  2. setInput debounces 150 ms and writes to localStorage.
  3. User runs the tool — click or ⌘/Ctrl + Enter → runActiveTool(input, options).
  4. Pure tool function in lib/json/ returns a discriminated union Result.
  5. Success → output + stats render in the right pane and status bar. Failure → editor gets gutter marker + span selection, status bar shows the error with a [ jump ] link.
  6. Output side effects: Copy uses navigator.clipboard; Download builds a Blob.

Result envelope

// lib/json/types.ts
type JsonError = {
  line: number; column: number;
  offsetStart: number; offsetEnd: number;
  message: string; hint?: string;
};

type Stats = {
  bytes: number;     // TextEncoder length of input
  keys: number;      // total unique key occurrences
  depth: number;     // max nesting (object + array)
};

type Result =
  | { ok: true;  output: string; stats: Stats }
  | { ok: false; error: JsonError; stats: null };

Tool implementations

FunctionBehavior
parse(input)Wraps JSON.parse. On SyntaxError, returns a normalized JsonError via lib/json/errors.ts.
beautify(input, { indent, sortKeys })Parse + JSON.stringify with indent. If sortKeys, walks the tree and rebuilds with sorted object keys (arrays untouched).
minify(input)Parse + JSON.stringify with no indent.
validate(input)Parse only; on success, walks the tree once to compute stats.
buildTree(input)For /parse. Returns a tree of nodes carrying type, key, raw value (leaves), and path string for copy.

All tools call parse first; that's the shared canonical entry point.

URL ↔ state binding

  • Path determines which tool.
  • Query params carry tool options (?indent=tab&sort=1). Option changes call history.replaceState; tab switches call history.pushState.
  • Hash is never used — input content stays in localStorage only.

localStorage shape

// key: 'jsontools:v1'
{
  "schemaVersion": 1,
  "input": "{ ... }",
  "tool": "beautify",
  "options": { "indent": 2, "sortKeys": false },
  "theme": "auto"
}

On read, validate the shape against a Zod-like check; on mismatch, discard. The schemaVersion lets us evolve the format without trapping users on a broken state.

6 · Error handling & edge cases

Parse error normalization

V8 / JavaScriptCore / SpiderMonkey each format SyntaxError.message differently. lib/json/errors.ts exposes normalizeJsonError(input, err):

  1. Try each engine's regex (V8: at position N; JSC: at line N column M; SpiderMonkey: at line N column M).
  2. If engine output is opaque, fall back to a tiny in-house JSON tokenizer that locates the first invalid token.
  3. Return { line, column, offsetStart, offsetEnd, message, hint? }.

Hint catalogue

Native message contains…Plain-English hint
Unexpected token 'Single quotes aren't valid JSON — use double quotes
Unexpected token }Trailing comma before }
Unexpected token ]Trailing comma before ]
Unexpected end ofInput ends mid-value — missing }, ], or "
Unexpected token (identifier-shaped)Keys must be quoted strings

Error presentation

  • Editor: red gutter marker on the error line; offending span gets a subtle red underline via CodeMirror decoration. Clicking the gutter marker scrolls to the line.
  • Status bar: › parse failed at 4:12 — trailing comma before } in red, with a [ jump ] link that scrolls and selects the span.
  • Right pane: muted placeholder — no output — with the error echoed below. We don't blank it.

Edge case behaviors

CaseBehavior
Empty or whitespace-only inputAction runs; muted › empty input. Not an error.
Valid scalar JSON (null, true, number, string)All valid. Stats: keys: 0, depth: 0. Tree shows a single leaf.
Input > 5 MBNon-blocking warning above editor.
Input > 50 MBHard confirm before running.
Non-text file uploadSniff for null bytes / UTF-16 BOM upfront; reject with friendly message.
File upload > 50 MBReject before reading.
Multiple files droppedUse first; notice naming the file used.
Non-file drag (image, URL)Ignored silently.
Sort keys on an arrayArrays positional — unchanged. Documented in option tooltip.
Duplicate keysNative parse keeps the last; surface a one-time muted notice Duplicate key 'foo' — kept last value.
Very deep nesting (> 1000)Catch Maximum call stack; report Input too deeply nested.
Numeric precision lossOpt-in detector flags as muted notice; off by default.
Clipboard write blockedToast: Couldn't copy — your browser blocked clipboard access.
LocalStorage quota / Safari privateSilent first time; on operation, set session flag to skip persistence and show one-time toast.

Deliberate non-features

  • No auto-format on type — beautify is one click.
  • No auto-validate while typing — CodeMirror's linter shows obvious gutter errors; explicit validate produces full stats.
  • No recovery of corrupted localStorage state — discard the whole entry.

Accessibility

  • Tabs are real <a> links, keyboard-navigable.
  • Action button has an explicit aria-label matching the current tool.
  • Status messages live in an aria-live="polite" region.
  • Color is never the only signal: › for status, "failed"/"valid" wording, focus rings on toggles.
  • Mono font at 14 px / 1.5 line-height; amber-on-charcoal and amber-on-paper both ≥ 4.5:1.

Manual QA checklist (pre-launch)

  1. Cold load on Safari iOS, paste 1 MB JSON, beautify → no jank.
  2. Drag-drop a 4 MB file in Firefox → output renders without freezing.
  3. Eight broken-JSON snippets → each shows the expected hint.
  4. Toggle theme three times rapidly → no flash.
  5. Full beautify cycle using only keyboard → no mouse needed.
  6. 320-px viewport → layout usable, no overflow.
  7. Private mode (no localStorage) → site still works.
  8. Screen reader on validate → success and failure both announce.
  9. Hard-reload twice with devtools open → CSS/JS cached, HTML fresh.
  10. Lighthouse on /beautify → ≥ 95 across all categories.
  11. Critical JS < 150 KB gz (verified via build report).
  12. CF Web Analytics shows the visit; privacy footer copy still accurate.

8 · Out of scope (v1)

  • JSON5 / JSONC lenient parsing.
  • JSON Schema validation.
  • Web Worker / streaming parser for > 50 MB payloads.
  • Share-via-URL or short-link sharing.
  • JSON ↔ YAML / CSV / TOML conversion.
  • Diff view between two JSON documents.
  • Auth, user accounts, saved snippets.
Each of these has its own reasoning in the brainstorm transcript. Revisit if real usage signals demand.
